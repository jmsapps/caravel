# Unreleased

## Added

- Added `CheckpointPlugin`, a first-party checkpoint capability providing
  durable committed-output evidence and checkpoint-backed selective execution.
  It requires an explicit `metadata_root` (no location is derived from the
  pipeline or run root) and stores schema-versioned records under
  `<metadata_root>/v1/checkpoints/<node_id>.json`. Records are invalidated and
  confirmed absent before output replacement, published and read back only
  after a successful save, and verified against the declaration and physical
  output before every reuse verdict. Removing the plugin leaves records inert:
  bare core ignores them and rejects checkpoint-dependent selection.
- Added `OwnershipPlugin`, a first-party ownership capability that prunes
  stale cross-run Caravel output using durable recorded ownership instead of
  filename conventions. It requires an explicit `metadata_root` and records a
  schema-versioned bound-plan inventory per pipeline at
  `<metadata_root>/v1/inventories/<pipeline>.json`. Deletion candidates come
  only from a prior valid inventory, must stay inside the managed pipeline
  root, and are removed idempotently before the new inventory is committed
  last. `stage_root` trees are never pruned, selective runs neither prune nor
  replace the inventory, foreign files and matching-looking directories
  survive, and interrupted reconciliation converges on rerun. Removing the
  plugin disables historical pruning without weakening Dataset replacement.
- Added `RunEvidencePlugin`, a first-party observer plugin that records one
  immutable schema-versioned event object per committed lifecycle transition
  under a required explicit `metadata_root`
  (`<metadata_root>/v1/events/<run_id>/<seq>-<kind>.json`) plus a regenerable
  summary derived only from events. Events carry identifiers, node facts, and
  exception class names — never payloads, credentials, storage options,
  environment values, or parameter values. `criticality='required'` (default)
  makes an evidence failure a distinct operational failure without
  invalidating committed checkpoints; `'best_effort'` surfaces failures in
  `RunResult.best_effort_errors` without changing execution. Corrupt or
  missing events fail summary regeneration explicitly and never affect
  checkpoint reuse.
- Added `LeasePlugin`, a first-party run-guard plugin providing advisory
  writer-conflict evidence, heartbeats, and abandoned-run diagnostics. It
  requires an explicit `metadata_root` and keeps one lease object per pipeline
  (`<metadata_root>/v1/leases/<pipeline>.json`) with holder identity, host,
  process ID, start time, and heartbeat time refreshed by a plugin-owned
  daemon thread that never executes user code. An apparently live lease
  refuses startup before user code runs; a stale lease is recovered with a
  durable recovery evidence object naming the abandoned run; a malformed
  lease fails closed for operator inspection; a mid-run heartbeat failure
  surfaces at teardown as a distinct operational failure; and normal shutdown
  releases the lease with no abandoned worker thread. The lease is advisory
  diagnostics — not a distributed lock — because generic fsspec has no
  portable atomic create-if-absent; the external scheduler remains the
  authoritative writer-serialization and hard-timeout contract.
- Added the singleton `OwnershipCapability` plugin protocol with immutable
  `OwnershipContext`/`PlanOutput` bound-plan facts, invoked by the executor
  once per run before any node executes.
- Added checkpoint error types to `caravel`: `CheckpointError`,
  `CheckpointIntegrityError`, `UnsupportedCheckpointVersionError`, and
  `CheckpointCommitUncertainError`; `UnsupportedCapabilityError` is now
  exported.
- Added `physical_partition_keys(dest)` and `verify_physical_output(dest,
  partition_keys)` to the six built-in Datasets so a checkpoint capability can
  record and verify physical output without class-name dispatch. Custom
  Datasets without this support still save on full runs but receive no
  checkpoint evidence and cannot serve as checkpoint-backed selective
  boundaries.
- Added deterministic logical plan compilation with immutable nodes, stable
  index-based node IDs, explicit input references, branch expansion, and
  byte-stable snapshots.
- Added runtime binding as the single owner of run roots, selectors, resolved
  output paths, and capability requirements.
- Added one executor path for linear steps, branch fan-out, route steps, and
  merge nodes.
- Added explicit runner plugins through `run(..., plugins=[...])` and
  `make_cli(..., plugins=[...])`.
- Added typed observer, run-guard, and singleton checkpoint-policy
  capabilities with deterministic ordering and failure semantics.
- Added immutable checkpoint callback contexts containing complete run and node
  facts. Stateful plugins own and require their storage configuration.
- Added `load_from(dest)` to built-in Datasets for capability-authorized loading
  of saved output.
- Added payload, parameter, credential, path, and plugin lifecycle security
  tests.

## Changed

- `StepContext.run_id` is now a UUID4 hex value generated for each invocation.
- Built-in Datasets validate complete payloads before destructive output
  replacement. JSON validation includes actual encoding, so unencodable values
  preserve prior output.
- Persisted built-in output replaces the complete Caravel-owned step directory,
  preventing stale partitions after a fewer-key rerun.
- `clean_dirs=True` without `stage_root` cleans only the resolved stage
  directory instead of the entire run root.
- Full runs pass outputs in memory across stage boundaries instead of reloading
  them from disk.
- Selective requests that require skipped historical output fail during
  binding unless a checkpoint capability is configured. Bare core never trusts
  raw file existence as completion evidence.
- Partitioned output directories are data-only; `.caravel_empty` is no longer
  written. Empty output on storage without durable directories requires
  external committed evidence to be reusable.
- Pipeline, stage, step, branch, route, partition, and plugin identifiers are
  validated before execution.
- `step(output=..., persist=...)` and `StepContext.persist` remain supported.

## Fixed

- Fixed stale partition files surviving a persisted rerun with fewer keys.
- Fixed `clean_dirs=True` deleting prior stages or unrelated run-root content.
- Fixed steps following a branch in the same stage receiving the pre-branch
  payload instead of the merged branch output.
- Fixed CLI parameter parse errors echoing raw parameter values.
- Fixed persisted-step logs claiming checkpoint evidence was written when no
  checkpoint capability was configured.
